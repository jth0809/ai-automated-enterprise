# Tracker Phase 2 운영 런북

## 목적과 경계

이 런북은 Phase 2 이후 무인 파이프라인을 사람이 감독하는 주간·월간·분기
루틴을 절차화한다. 관제(WP2.4)·골든셋(WP2.3)·정식 검수 UI(WP2.5)·로컬
임베딩 병합(WP2.6)이 만든 신호를 읽고, 상태 동결을 안전하게 해제하는
방법을 규정한다.

- 모든 관리자 조작은 토큰 게이트 콘솔(`운영 상태`·`검수 큐` 섹션)로만
  수행한다. 토큰은 브라우저 메모리와 요청 헤더에만 두고 localStorage·URL·
  로그·DOM에 남기지 않는다.
- production에서 수동 SQL 변경이나 `kubectl apply`를 하지 않는다. 배포와
  비상 중지는 `gitops/` 커밋과 Flux reconcile만 사용한다.
- 골든 평가는 API 키가 없으면 `OFFLINE_REPLAY`만 실행하고 그 결과를 LIVE
  baseline이나 실제 모델 품질로 표시하지 않는다. LIVE 활성화는 별도로
  기록한다.
- 외부 기사 원문·인용문·바이너리를 추가로 저장하지 않는다. 새 egress가
  필요한 미래 모델은 CNP `toFQDNs` allowlist와 보안 검토 없이 활성화하지
  않는다.

## 콘솔 읽는 법

`운영 상태` 섹션(`OpsPanel`)은 다음을 노출한다.

- **동결 배지**: `Frozen`이면 사유·트리거·시각이 함께 뜬다. `Active`면 상태
  갱신이 정상 진행 중이다.
- **최신 골든 실행**: 모드(`LIVE`/`OFFLINE`/`DRILL`), 상태, 합의율,
  매칭/전체 수. 모드가 `LIVE`가 아니면 그 합의율을 품질 기준선으로 인용하지
  않는다.
- **관리도 지표**: 지표코드·상태·값. 위반 항목은 강조된다.
- **피드 데드맨**: 전체 대비 경보 피드 수와 상태.

## 주간 루틴 (1~2시간)

1. **대기 검수 우선순위 처리.** `검수 큐`에서 `Pending` 탭을 우선순위
   내림차순으로 검토한다. TRL/EGL 8~9 전진, 레벨 점프, 플루크 불일치,
   도달 후보, 서킷 브레이커 사유를 먼저 본다.
2. **승인/반려 기준.** 인용이 원문 부분 문자열과 일치하고 사건 유형·검증
   수준 도출이 루브릭과 맞으면 승인한다. 인용 불일치, 과장된 레벨, 단일
   Tier 3 근거는 반려한다. 결정은 사유와 함께 남긴다.
3. **골든 리포트 확인.** `운영 상태`의 최신 골든 실행 합의율을 본다. 모드가
   `OFFLINE`/`DRILL`이면 회귀 감시 용도로만 읽고 LIVE 품질로 보고하지
   않는다.
4. **데드맨 경보 확인.** 경보 피드가 있으면 소스 상태를 점검한다. 침묵이
   중위 간격의 2배를 넘으면 피드 설정·egress·소스 자체를 조사한다.
5. **동결 여부 확인.** `Frozen`이면 아래 "상태 동결 해제" 절차를 따른다.

## 월간 루틴 (1시간)

1. **관리도 baseline·오경보 검토.** 최근 위반과 연속 위반 수를 보고, 정상
   변동을 위반으로 잡는 baseline이 있으면 다음 재보정 후보로 기록한다(수식·
   파라미터 변경은 WP4.2에서 다룬다).
2. **소스 상태와 비용.** 피드 커버리지와 일일 LLM 비용 가드 여유를 점검한다.
   비용이 상한에 근접하면 게이트 재현율과 오분류 루프를 확인한다.
3. **검수 backlog.** `Pending` 총계 추세를 확인한다. backlog가 누적되면
   우선순위 임계나 주간 시간 배분을 조정한다.

## 분기 루틴 (2~4시간)

1. **50-케이스 재합의와 드리프트 드릴.** 골든셋 50건을 재라벨링해 합의
   기준선을 갱신하고, `tracker-circuit-breaker-drill.md` 절차로 동결→해제
   훈련을 1회 수행한다.
2. **버전 변경 검토.** 프롬프트·모델·루브릭·스키마 버전 스탬프를 점검한다.
   모델 은퇴가 예정되면 골든 재검증 절차로 마이그레이션을 흡수한다.
3. **해제/롤백 복습.** 상태 동결 해제와 필라 6 `제도 후퇴(rollback)` 처리
   절차를 복습한다.

## 상태 동결 해제

상태 갱신이 동결되면 갱신은 멈추지만 수집·분류·검수 기록은 계속된다.

1. `운영 상태`에서 동결 사유·트리거·시각을 확인하고, 골든 합의율과 관리도
   위반이 정상으로 회복됐는지 검증한다. 회복은 실제 원인 해소(예: 골든
   재실행 합의율 회복)로 확인하고, 지표만 보고 성급히 해제하지 않는다.
2. 회복이 확인되면 `Release reason`에 1~2,000자 사유를 적고 `Release freeze`를
   누른다. 성공 시 감사 로그가 남고 토큰은 기록되지 않는다.
3. 아직 동결 상태가 아니면 해제는 409로 거부된다. 다른 관리자가 먼저
   해제했는지 콘솔을 새로고침해 확인한다.
4. 해제 후 `검수 큐`를 다시 로드해 동결 중 보류된 결정이 정상 반영되는지
   확인한다.

## 비밀·배포 원칙

- `ANTHROPIC_API_KEY`, `TRACKER_ADMIN_TOKEN`, `TRACKER_FEEDS` 등은 OCI Vault
  경로에 두고 External Secrets Operator로 주입한다. 평문 커밋·이 문서 기재를
  금지한다.
- 관리자 토큰은 콘솔 세션 메모리에만 존재한다. 브라우저를 닫으면 사라지며
  영속 저장하지 않는다.
- 모든 인프라 변경은 `gitops/` 커밋 → Flux reconcile로만 반영한다. 골든
  임계 미달·관리도 위반 시 상태 갱신은 자동 동결되고 인간이 해제한다.
- 새 RSS 피드·외부 API를 붙이려면 도메인별 CNP `toFQDNs` allowlist를 먼저
  등록하고 보안 검토를 거친다. 로컬 임베딩 병합(WP2.6)은 외부 API·모델
  파일·vector DB·새 egress를 사용하지 않으므로 추가 등록이 필요 없다.
